# SA + SD 混合文件範本

起草或修訂最終混合文件、補上 SD 章節與合併驗收條件時，讀取本文件。

本範本**不是僵硬規定**。此 skill 不強制 SD 必須包含哪些固定章節。請先理解原 `issue-sd-doc-planner` skill 的設計理念（精要的閱讀導覽、設計摘要、明確核心決策、具體資料/介面/程式結構、實作注意事項、待釐清問題、技術驗收），再依現有 SA 內容、專案情境與可讀性，自行決定最適合的章節安排。章節數量應受控制，避免因簡單追加 SD 而膨脹。

## 撰寫風格

- 預設以繁體中文撰寫。
- 文件對象同時包含架構審閱者與工程實作者：前半段讀 SA 架構脈絡，後半段讀 SD 系統設計。
- 一致使用來源標記（`SA-confirmed`、`Code-confirmed`、`Doc-supplement`、`Inferred`、`Design-decision`、`Unknown`），讓每項設計陳述可追溯。
- 結構化內容優先使用帶有 `來源` 欄位的 Markdown 表格；若僅以表格呈現流程、互動或資料流仍不夠清楚，可搭配 Mermaid 流程圖輔助說明。
- SD 要具體到足以進行工程審查與實作規劃，但不要寫成 coding plan、commit plan 或逐行程式修改說明，也不要複製其他專案的資料表名稱、頁面名稱或業務規則。
- 若 SA、程式碼與補充資料未能提供某面向所需的足夠細節，寫入 `目前 SA、程式碼與補充資料無法確認`，並將該列標記為 `Unknown`，同時於「待釐清事項」登錄。

## 混合文件的整體結構（概念，非固定章節清單）

最終文件由三段構成：

```text
# <功能名稱> SA + SD 文件

## 文件資訊
## 閱讀導覽          ← 說明本文件前段為收斂後 SA、後段為 SD，何處為邊界/背景
## 來源標記說明

── 前半段：收斂後的 SA 架構脈絡 ──
（保留必要的背景與目標、範圍、角色與權限、功能需求、業務規則、流程、
  架構概觀、資料與整合概念等；已收斂、去重、合併，仍保留來源標記）

── 後半段：承接 SA 的 SD 系統設計 ──
（依實際情境選用下方 SD 建構區塊，無關者省略、需合併者合併）

── 文件尾段 ──
## 待釐清事項        ← 必須保留
## 驗收條件          ← SA 與 SD 整合後的總驗收條件，必為最後一章
```

`閱讀導覽` 緊接 `文件資訊`；`來源標記說明` 緊接 `閱讀導覽`。前半段與後半段必須連貫，不得重複堆疊同一內容。

### 資料設計（涉及 persisted data 時）

> 若不影響 persisted data、schema、SQL、storage、reporting、import/export 或資料生命週期，明確寫出無 schema change，並說明資料來源與查詢策略。涉及時遵循 `data-design-checklist.md`。

實際 persisted 資料表使用 TableSchema 格式（SQL/storage 型別，非 .NET runtime 型別）：

| 欄位 | 型別建議 | 必填 | Key / Index | 說明 |
| --- | --- | --- | --- | --- |
| `<欄位>` | `<SQL / storage 型別>` | <是 / 否> | <PK / FK / unique / index / default / -> | <用途、來源、生命週期或限制> |

- 單欄位的 PK/FK/unique/index/default 放在 `Key / Index` 欄位。
- `額外索引` 子節：只放第一版必要索引（schema 變更交付時就必須存在、支援核心查詢/寫入/唯一性/一致性或明確第一版效能需求）。
- `後續評估索引` 子節：只放有合理依據但可延後的索引（reporting/export、歷史查詢、非核心過濾排序，待 slow query、資料量成長或使用頻率再評估）。
- 不列非必要或推測性索引。

## 驗收條件（最後一章，SA + SD 整合）

原 SA 文件中的驗收條件**不得**留在 SA 區段中分散呈現。完成收斂與 SD 補充後，將 SA 原有驗收條件與 SD 新增驗收條件整合，統一移到文件**最後一章**，作為整份混合文件的總驗收條件。

合併原則：
- 移除重複或語意高度重疊的驗收條件。
- 將過度細碎的驗收條件收斂成核心、明確、可判斷的項目。
- 同時涵蓋 SA 層面的架構正確性與 SD 層面的系統設計完整性。
- 不保留彼此衝突的驗收條件；若衝突無法判斷，列入「待釐清事項」並向使用者確認。
- 驗收條件放在文件最後，不散落各章節。
- 以「夠用即止」為原則，不膨脹成測試案例清單或實作 checklist。目標是讓 reviewer 能快速判斷這份最終文件是否足以支撐後續實作。

| 編號 | 驗收條件 | 層面 |
| --- | --- | --- |
| AC-01 | <可驗證的架構正確性或系統行為結果> | SA / SD |
| AC-02 | <可驗證的資料、介面、權限、流程或一致性結果> | SD |
