# SA + SD Document Template

起草或修訂最終 SA + SD 混合文件時，請使用本模板。本文件同時服務兩類讀者：**進行需求與設計審查的人**，以及**依此文件進行實作的工程師**。SA 部分說明架構脈絡，SD 部分具體到足以進行工程審查與實作規劃。

## Required Style

- 預設以繁體中文撰寫。
- SA 內容維持**架構層級**：說明要做什麼、由哪些組件與職責構成、彼此如何互動、受哪些規則與邊界約束。
- SD 內容深入到**實作層級**：資料表結構、API 合約、模組職責邊界、流程細節、權限檢查位置、錯誤處理方式等。
- **不**寫成 coding plan、逐行修改說明或 commit 計畫。
- 保留來源標記，讓審查者理解每個陳述的依據。一致使用：`PRD-confirmed`、`Code-confirmed`、`SA-confirmed`、`Doc-supplement`、`Inferred`、`Design-decision`、`Unknown`。
- 結構化內容優先使用含 `來源` 欄位的 Markdown 表格，避免在有表格來源欄時仍逐句重複標記。
- `來源標記說明` 緊接 `文件資訊` 之後。
- 流程以 Mermaid flowchart 表示主線；架構圖以 Mermaid graph 呈現組件關係與互動。
- 除標點符號外，所有特殊符號（`/`、`+`、`-`、`...` 等）一律使用半形。
- 各章節同時涵蓋 SA 與 SD 內容時，SA 層級在上、SD 層級在下，以小標題或說明文字區隔，不重複描述同一件事。
- 必須保留 `待釐清事項`；若無未解問題，保留章節並填入一列 `目前無待釐清事項`。
- 若現有資料未提供足夠資訊以填寫某章節，寫入 `目前 SA、程式碼與補充資料無法確認`，並在適用時將該列標記為 `Unknown`。
- SD 章節依實際情境選用；無關者省略，不為套模板而產生空洞章節。

## Section Guidance

- `文件資訊`：metadata，保持簡潔。
- `來源標記說明`：除非使用者明確要求移除，否則始終置於文件前段。
- `功能概述`：說明功能為何存在、支援什麼使用情境、核心職責是什麼。若有相關但不在本次範圍的功能或後續規劃，在此說明其邊界定位。多個目標時用表格。
- `角色與權限`：以行為層級描述，不寫實作細節。放在範圍之前，讓讀者在閱讀需求與範圍前先掌握角色全貌。
- `範圍`：明確區分包含與不包含。
- `需求與業務規則`：FR 描述系統要做什麼，以 FR 編號分組；BR 描述驗證、狀態、計算、可見性、資料來源等約束，並說明違反時的處理方式。
- `流程`：Mermaid flowchart 同時涵蓋業務步驟與實作關鍵節點，不需拆成兩張圖。資料流說明表格補充各節點的資料來源、驗證與輸出細節；流程單純時可省略。
- `架構與模組設計`：架構圖與組件說明為 SA 層級；模組職責與介面合約為 SD 層級。SA 層級在上，SD 層級在下，不重複描述同一件事。模組職責表格內若有替代方案被排除，在對應列說明未採用原因，不需獨立設計決策章節。
- `資料設計`：概念模型為 SA 層級，說明業務實體與關係；Schema 設計為 SD 層級，使用 SQL / storage 型別。索引分為第一版必要索引與後續評估索引，不列推測性索引。不涉及 schema change 時明確說明並省略 Schema 設計子節。
- `實作注意事項`：將錯誤處理、冪等、一致性、效能、migration、rollout / rollback、concurrency、權限稽核設計與可觀測性整合於此，不分散成多個淺層章節。
- `待釐清事項`：必須保留。修訂時把已回答列移至對應章節並更新來源標記；無剩餘時保留一列 `目前無待釐清事項`。
- `驗收條件`：必為文件最後一章，作為整份文件的總驗收條件。把 SA 原有驗收條件與 SD 新增驗收條件整合、去重、收斂成核心可判斷的項目，不散落各章節、不膨脹成測試案例清單。同時涵蓋 SA 架構正確性與 SD 設計完整性；衝突且無法判斷的條件列入「待釐清事項」確認。此章節供下游 `dev-acceptance-verifier` 逐條驗收。
