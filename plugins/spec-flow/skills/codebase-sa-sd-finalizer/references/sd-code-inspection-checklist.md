# 程式碼查找清單（既有專案）

對既有專案查找與 SA 文件相互對照的現有程式碼時，先讀取本文件。優先使用程式碼知識圖譜查找工具，而非盲目開檔；程式碼知識圖譜查找不可用或查無結果時，退而使用一般程式碼搜尋與檔案閱讀，但仍維持「對照 SA、擷取設計影響」的原則。查找與文件收斂、SD 設計同時進行。

## 查找目標（與 SA 互相對照）

針對 SA 文件提及的內容，在程式碼中查找對應現況：
- SA 中提及的模組、服務、流程或資料來源。
- 既有 API、controller、service、repository、model、job、event、queue、config。
- Route、command、event handler、排程任務與背景工作的進入點與負責元件。
- 既有命名慣例、資料結構、DTO/ViewModel、request/response 型別、SQL 與資料表參考。
- 既有錯誤處理、retry、驗證、concurrency 與邊界情況處理方式。
- 既有權限檢查、ownership、資料可見範圍、角色對應與稽核/logging 機制。
- 既有測試、fixture、seed、feature flag、migration 與相容性處理。
- 與本次設計相關的上下游依賴與外部整合設定。
- SA 文件中提到、但程式碼中可能已有既有實作或限制的部分。

## 應捕捉的發現

只捕捉影響系統設計的事實：
- 目前的進入點與負責元件、責任邊界。
- 既有資料模型、SQL 結構、資料表關聯與資料所有權。
- 既有 API / 介面 / event 合約與整合行為。
- 既有角色、權限檢查、ownership、核准流程、稽核與 logging 行為。
- 既有錯誤處理、retry、驗證、concurrency 與邊界情況。
- 既有 export/import 模式、檔案命名與資料整形邏輯。
- 可作為實作模式參考的現有模組，以及為何適合沿用。
- 影響實作信心的測試覆蓋缺口。

不要僅憑命名相似就認定為同一事物。若存在多個候選元件，檢查足夠上下文以確認最可能的負責方，或將此模糊點記錄到「待釐清事項」。

## SA 與程式碼不一致時

若既有程式碼與 SA 文件描述不一致，依「核心行為規範」**暫停並向使用者確認**，不得自行判定以哪一方為準。SA 是主要事實來源，但程式碼現況可能揭示 SA 未涵蓋或已過時的限制——這類落差需由使用者裁決，不可在文件中悄悄以某一方覆蓋另一方。

## 輸出期望

- 將觀察轉化為**設計影響**，而非只是列出檔案。
- 只在有助於工程審查時，才列出精確的路徑、類別、方法、資料表或 enum 名稱。
- 在適用時，指出應以哪個現有模組作為實作模式。
- 將缺乏實作支撐的情況記錄為設計缺口或風險。
- 不臆造資料庫欄位、API 合約、權限或業務規則；推斷而來者標記為 `Inferred` 並說明依據。
- 不改寫屬於 SA 的業務範圍。

建議表格格式：

```text
| 來源 | 項目 | 程式位置 / 既有模式 | 與 SA 的對照 / 設計影響 |
| --- | --- | --- | --- |
| Code-confirmed | <目前行為或模式> | <路徑、類別、方法、資料表或元件> | <與 SA 一致/落差，以及對本設計的影響> |
```
