# SA Document Template

起草或修訂最終 SA Markdown 文件時，請使用本模板。本 SA 同時服務兩類讀者：**進行需求審查的人**，以及**後續依此文件獨立完成 SD 的 AI**。撰寫時兼顧兩者：敘述精煉好讀，資訊完整可被接手。

## Required Style

- 預設以繁體中文撰寫獨立的系統分析文件。
- 嚴格維持**架構層級**：說明要做什麼、由哪些組件與職責構成、彼此如何互動、受哪些規則與邊界約束。
- **不**包含 SD 細節：資料庫 schema、API 合約 / 欄位定義、route 與命名規則、migration / backfill 計畫、逐方法實作步驟、套件選型與目錄結構。資料與介面僅描述到「概念模型 / 概念整合點」。
- **不**使用「由 SD 設計」「供 SD 參考」「留待 SD 補充」等轉嫁語句。不屬於 SA 的內容直接不寫。
- 保留來源標記，讓審查者理解每個陳述的依據。一致使用：`PRD-confirmed`、`Code-confirmed`、`Inferred`、`Design-decision`、`Unknown`。
- 結構化內容優先使用含 `來源` 欄位的 Markdown 表格，避免在有表格來源欄時仍逐句重複標記。
- `來源標記說明` 緊接 `文件資訊` 之後。
- 背景說明用短段落；範圍、角色、欄位、業務規則、架構組件、待釐清事項用表格。
- 流程以 Mermaid flowchart 表示主線；架構圖以 Mermaid graph 呈現組件關係與互動。
- 除標點符號外，所有特殊符號（`/`、`+`、`-`、`...` 等）一律使用半形。
- 必須保留 `待釐清事項`；若無未解問題，保留章節並填入一列 `目前無待釐清事項`。
- 若 PRD 與程式碼未提供足夠資訊以填寫某章節，寫入 `目前 PRD 與程式碼未提供足夠資訊`，並在適用時將該列標記為 `Unknown`。

```text
僅列沒有任何單一模組能獨佔、必須跨模組或端對端才能驗證的條件。單一模組能獨佔的驗收條件不放這裡（模組 SA 不含驗收條件，由下游 SD 補上）。

| 編號 | 驗收條件 |
| --- | --- |
| AC-01 | <跨越多個模組、沒有任何單一模組能獨佔的端對端驗收條件> |
```

母文件撰寫要點：
- 只放**整體**與**跨模組**內容；單一模組能獨佔的需求、規則、流程、驗收條件放該模組 SA，不在母文件重複。
- `模組 / 功能分解` 是母文件的核心：清楚交代每個模組的範圍、彼此依賴與先後順序、對應檔名，使各模組可獨立接手。
- 共用的概念實體或契約，明確指派一個「擁有者模組」或「最先做的基礎模組」，避免兩個模組各寫一份而衝突。
- `跨模組／端對端驗收條件` 是母文件獨有的驗收章節：母文件不往下游 finalize（不會有 `-sd.md`），故跨模組與端對端的驗收條件只能落在母文件這裡，供下游 `dev-acceptance-verifier` 的母文件驗收使用。
- 母文件是**活文件**：分解於開發過程被修訂時就地更新，並同步調整受影響的模組 SA。

## Section Guidance

- `文件資訊`：metadata，保持簡潔。拆分時，模組 SA 在此加一列「所屬母文件 | `<feature>-overview-sa.md`」指回母文件。
- `來源標記說明`：除非使用者明確要求移除，否則始終置於文件前段。
- `功能概述`：說明功能為何存在、支援什麼使用情境、核心職責是什麼。若有相關但不在本次範圍的功能或後續工單，在此說明其邊界定位。多個目標時用表格。
- `角色與權限`：以行為層級描述，不寫實作細節。放在範圍之前，讓讀者在閱讀需求與範圍前先掌握角色全貌。
- `範圍`：明確區分包含與不包含。
- `需求與業務規則`：FR 描述系統要做什麼，以 FR 編號分組；BR 描述驗證、狀態、計算、可見性、資料來源等約束，並說明違反時的處理方式。
- `流程`：以 Mermaid flowchart 呈現主線，異常路徑以條列補充，每個流程後附簡短來源標記。
- `架構概觀`：交代受影響組件、職責、互動與邊界；既有專案以 GitNexus 現況為依據。架構圖使用 Mermaid graph 並在箭頭上標註觸發情境或傳遞內容；組件細節用組件說明表格補充；組件間的互動情境用主要互動說明表格補充。
- `資料與整合概念`：僅到概念層級 - 有哪些業務實體、彼此關係、對外整合對象。
- `待釐清事項`：必須保留。修訂時把已回答列移至對應章節並更新來源標記；無剩餘時保留一列 `目前無待釐清事項`。
