# SA Document Template

起草或修訂最終 SA Markdown 文件時，請使用本模板。本 SA 同時服務兩類讀者：**進行需求審查的人**，以及**後續依此文件獨立完成 SD 的 AI**。撰寫時兼顧兩者：敘述精煉好讀，資訊完整可被下游接手。

## Required Style

- 預設以繁體中文撰寫獨立的系統分析文件。
- 嚴格維持**架構層級**：說明要做什麼、由哪些組件與職責構成、彼此如何互動、受哪些規則與邊界約束。
- **不**包含 SD 細節：資料庫 schema、API 合約／欄位定義、route 與命名規則、migration/backfill 計畫、逐方法實作步驟、套件選型與目錄結構。資料與介面僅描述到「概念模型 / 概念整合點」。
- **不**使用「由 SD 設計」「供 SD 參考」「留待 SD 補充」等轉嫁語句。不屬於 SA 的內容直接不寫。
- 保留來源標記，讓審查者與下游 AI 理解每個陳述的依據。一致使用：`PRD-confirmed`、`Code-confirmed`、`Inferred`、`Design-decision`、`Unknown`。
- 結構化內容優先使用含 `來源` 欄位的 Markdown 表格，避免在有表格來源欄時仍逐句重複標記。
- `閱讀導覽` 緊接 `文件資訊`；`來源標記說明` 放在 `閱讀導覽` 之後。
- 背景與脈絡用短段落；範圍、角色、欄位、業務規則、架構組件、待釐清事項與驗收條件用表格。
- 必須保留 `待釐清事項`；若無未解問題，保留章節並填入一列 `目前無待釐清事項`。
- 若 PRD 與程式碼未提供足夠資訊以填寫某章節，寫入 `目前 PRD 與程式碼未提供足夠資訊`，並在適用時將該列標記為 `Unknown`。

## 母文件（拆分總覽 SA）Template

當「模組／功能分解」定案為**拆分**時，除了各模組各自的標準 SA（上方結構）外，另產出一份母文件作為整組文件的地圖。母文件不往下游 finalize／build，只承載整體脈絡與跨模組資訊。

```text
# <功能名稱> 拆分總覽 SA（母文件）

## 文件資訊

| 項目 | 內容 |
| --- | --- |
| 功能名稱 | <功能名稱> |
| 文件性質 | 拆分總覽母文件（不單獨往下游 finalize／build） |
| PRD 來源 | <文件 / 網址 / Issue> |
| 專案類型 | <既有專案 / 全新專案> |
| 模組數 | <N> |

## 閱讀導覽

本文件為「<功能名稱>」拆分後的總覽母文件，說明整體背景、模組分解與彼此關係。各模組的詳細 SA 見對應模組文件；實作時每個模組各自獨立流過 SD → 計畫 → 分支。

## 來源標記說明

<沿用標準 SA 的來源標記表>

## 背景與目標

<整體層級的背景與目標>

## 名詞定義

<跨模組共用術語>

## 範圍

<整體包含 / 不包含>

| 來源 | 模組 / 功能 | 涵蓋範圍（FR / 職責） | 依賴 / 先後順序 | 對應模組 SA 檔 |
| --- | --- | --- | --- | --- |
| Design-decision | <模組名> | <該模組負責什麼> | <須先完成的模組；最先做填「無」> | `<feature>-<module-slug>-sa.md` |

```

母文件撰寫要點：
- 只放**整體**與**跨模組**內容；單一模組能獨佔的需求、規則、流程、驗收條件放該模組 SA，不在母文件重複。
- `模組／功能分解` 是母文件的核心：清楚交代每個模組的範圍、彼此依賴與先後順序、對應檔名，使下游能逐模組獨立接手。
- 共用的概念實體或契約，明確指派一個「擁有者模組」或「最先做的基礎模組」，避免兩個模組各寫一份而衝突。
- 母文件是**活文件**：分解於開發過程被修訂時就地更新，並同步調整受影響的模組 SA。

## Section Guidance

- `文件資訊`：metadata，保持簡潔。拆分時，模組 SA 在此加一列「所屬母文件 | `<feature>-overview-sa.md`」指回母文件。
- `閱讀導覽`：說明如何閱讀本 SA、哪些相關功能僅作為邊界或背景，以及本文件作為 SD 輸入的定位。
- `來源標記說明`：除非使用者明確要求移除，否則始終置於文件前段。
- `背景與目標`：說明功能為何存在、支援什麼結果；多個目標時用表格。
- `名詞定義`：收錄易混淆的領域術語，尤其狀態、角色、來源系統與業務物件。
- `範圍`：明確區分包含與不包含。
- `角色與權限`：以行為層級描述，不寫實作細節。
- `功能需求`：含多頁面／流程／能力時，用 FR 編號分組。
- `業務規則`：驗證、狀態、計算、可見性、資料來源等規則放此處。
- `流程`：編號步驟，每個流程後附簡短來源。
- `架構概觀（現況與影響）`：本文件供下游 SD 接手的關鍵章節。交代受影響組件、職責、互動與邊界；既有專案以 GitNexus 現況為依據。保持架構層級，不做 schema / API / 逐方法設計。
- `資料與整合概念`：僅到概念層級——有哪些業務實體、彼此關係、對外整合對象。不設計欄位、型別或合約。
- `待釐清事項`：必須保留。修訂時把已回答列移至對應章節並更新來源標記；無剩餘時保留一列 `目前無待釐清事項`。
- `驗收條件`：精簡收斂，僅列核心且可明確判斷的項目，「夠用即止」，不逐一窮舉細節。預設不加 `來源` 欄，來源可從前述章節追溯。
