# codebase-sa-builder

從使用者提供的 PRD（形式不限：文件、網址連結、Issue、規格、備忘等）出發，產出一份**架構層級**的獨立 SA（System Analysis）文件。若目標為既有專案，優先透過 GitNexus 查找相關現有程式碼，並在查找的同時進行文件設計；若為全新專案，直接進入文件設計流程。最終文件以繁體中文寫成 `.md` 檔案。

本 skill 的產出需同時滿足兩個目標：
- **易於人類閱讀與審閱**：結構清晰、敘述精煉、可作為需求審查與利害關係人討論的依據。
- **足以支撐下游 SD**：資訊完整且結構化，讓後續 AI 能僅依賴本 SA 文件，獨立完成 SD（System Design）設計。

需要細節時，請參考本 skill 內的 references：
- 起草或修訂最終 SA 文件前，請讀取 `references/sa-document-template.md`。
- 針對既有專案查找相關程式碼前，請讀取 `references/codebase-discovery.md`。
- 出現任何疑問或不明確之處時，請讀取 `references/clarification-and-open-questions.md`。

## 核心行為規範（最高優先，調用時必須遵守）

**遇到模糊需求必須暫停並反問，不得擅自假設或繼續設計。**

當本 skill 被實際調用執行、且設計過程中出現以下任一情況時，必須**立即停下來**向使用者提問，取得明確回覆後才繼續：

- 需求不明確、語意有多種解讀、或 PRD 與現有程式碼互相矛盾。
- 架構邊界不清晰：不確定某職責、組件、資料或流程是否屬於本次範圍。
- 角色、權限、業務規則、狀態流轉或預期行為無法從 PRD 或程式碼確認。
- 任何會實質改變範圍、架構、功能需求或驗收條件的判斷。

禁止以「先假設一個方向繼續做」「這部分等之後再說」的方式略過。若使用者在被詢問後選擇暫緩、跳過、或尚未達成共識，則該問題必須完整彙整至文件的「待釐清事項」章節，不得遺漏或省略。

## 規模判斷（最前置，於正式分析前反問）

本 skill 是管線入口，但**不是每個需求都該走完整 SA 流程**。讀完需求後、進入 GitNexus 查找或正式設計之前，先依下列三個判準判斷規模：

1. 是否會新增或調整**角色與權限**？
2. 是否會跨越目前**單一模組的職責邊界**（牽動其他模組）？
3. 是否需要新建或調整**既有架構假設**？

**無論你自己的判斷為何，一律先用選項反問使用者**（與「模組／功能分解」同一套反問模式，需要時可同一輪詢問）：

- 列出兩個明確選項：`完整 SA 流程` 與 `輕量開發文件`。
- 依判準標記**恰好一個** `(推薦)` 並置於第一項：符合任一判準傾向 `完整 SA 流程`；三者皆否傾向 `輕量開發文件`。推薦代表你的判斷，最終由使用者決定，使用者可透過自訂輸入覆蓋。

定案後：
- **完整 SA 流程**：照本 skill 既有流程繼續（含後續「模組／功能分解」的單一／拆分反問）。
- **輕量開發文件**：本 skill **不產出任何文件**，停下來告知使用者改調用 `codebase-dev-doc-builder`（產出一份 `<feature>-dev.md`，直接交給 `dev-planner`）。不自動調用，維持「純引導」慣例。

若使用者選輕量、但你在過程中已看出需求其實觸及上述任一判準，據實說明後仍以使用者選擇為準；必要的疑慮記入後續文件的「待釐清事項」。

## SA 與 SD 的界線（嚴格遵守）

本 skill 僅產出 **SA**，內容嚴格聚焦於**架構層面**。SA 回答「要做什麼、為什麼、由哪些組件與職責構成、彼此如何互動、受哪些規則與邊界約束」；SD 才回答「具體如何實作」。

不得寫入 SD 細節，包括但不限於：資料庫 schema 設計、API 合約／欄位定義、route 與命名規則、migration/backfill 計畫、逐類別或逐方法的實作步驟、套件選型與目錄結構。

**內容應自然收斂於 SA 該涵蓋的範疇。** 不得出現「此部分由 SD 設計」「供 SD 參考」「留待 SD 補充」等將責任推給下游文件的備註語句——若某內容不屬於 SA，就直接不寫，而非寫入並標記。SA 對資料與介面只描述到「概念模型 / 概念整合點」的層級（有哪些業務實體、彼此關係、對外整合對象），不展開成具體設計。

## 模組／功能分解（拆分決策，於 SA 發起）

把需求拆成幾個模組／功能屬於**架構分解**，因此在 SA 階段決定。下游每個模組各自獨立流過整條管線（SD → plan → branch → build），形成 **1 模組 SA = 1 `-sd.md` = 1 實作計畫 = 1 分支 = 多 commit** 的對應；分支只是模組往下游的載具，不是另一個獨立決策。

**收斂優先：** 只在需求確實可分解、且分開交付有實益時才拆；規模不大時不硬拆。拆分提案要像模組一樣聚焦、可獨立交付，不為湊數而拆。

**一律反問，不得自行定案。** 完成初步 PRD 分析後，即使你判斷單一最優，也要用選項詢問使用者後才決定文件結構：

- `單一`（整份需求一份 SA，維持既有單檔流程）一律列為選項之一。
- 依你的判斷，再列出**最多 3 個**具體拆分提案，每個提案列出它分成哪幾個模組／功能及各自範圍。給幾個（或不給）由你決定，以收斂為原則。
- **恰好一個選項標記 `(推薦)` 並置於第一項**（如 `(推薦) 單一` 或 `(推薦) 拆為 3 模組：…`）。推薦代表你的最優解，**不一定是拆分**：你可同時推薦「單一」並另列拆分備選，此時 `(推薦)` 留在「單一」。
- `使用者自決`（自行決定切法）透過提問內建的自訂輸入提供，不另佔一個選項格。
- 提問至少需 2 個明確選項；若你判斷單一最優而未提任何拆分，仍至少列出一個你認為最值得對照的拆分提案供權衡。

定案後：
- **選單一**：照既有流程輸出一份 SA 文件。
- **選拆分**：輸出 `1 份母文件（拆分總覽 SA）+ N 份模組 SA 文件`（結構見 `references/sa-document-template.md`）。母文件是「地圖」，不往下游 finalize／build；各模組 SA 才是獨立往下游的單位。

**母文件是活文件（即時回寫）：** 分解可被後續修訂。若下游在 finalize 或開發某模組時發現分解該調整，回到本 skill 更新母文件；**當前模組本輪冒出、會影響其他模組的調整，應於本輪即回寫母文件**（必要時連同受影響的模組 SA 一起調），使後續模組開工時讀到最新狀態。

## Input

請提供下列其中一項作為 PRD：
- 直接在 prompt 中提供完整需求內容
- 需求文件、規格、備忘或其節錄
- 網址連結（必要時讀取其內容）
- Issue 內容（可直接貼上，或提供 `project_id` 與 `issue_number` 以透過 `gitlab-issue-fetch` 取得）
- 待修訂的現有 SA Markdown 檔案

選填：
- 是否為既有專案 / 全新專案（未說明時自行判斷：能透過 GitNexus 查到相關 repo 與程式碼即視為既有專案）
- 輸出檔案名稱、輸出目錄、輸出語言覆蓋設定

未提供輸出路徑時：
- 將 Markdown 檔案寫入目前工作目錄。
- **單一**：有 issue 編號時使用 `issue-<issue_number>-sa.md`；否則使用 `sa.md`。
- **拆分**：建立一個資料夾收納整組文件，母文件命名為 `<feature>-overview-sa.md`，各模組命名為 `<feature>-<module-slug>-sa.md`（有 issue 編號時以 `issue-<issue_number>` 作為 `<feature>` 前綴）。各模組 SA 在 `文件資訊` 標明所屬母文件，母文件在分解章節標明各模組對應檔名，使整組文件可互相追溯。

預設輸出語言：
- Skill 指令與內部推理可保留英文。
- 最終 SA Markdown 文件預設以繁體中文撰寫，除非使用者明確指定其他語言。
- 釐清問題預設以繁體中文詢問。

## Workflow

1. 取得並通讀 PRD。內容不完整且提供了 `project_id` 與 `issue_number` 時，先呼叫 `gitlab-issue-fetch`；提供網址時，必要時讀取連結內容。
2. 依「規模判斷」用選項反問使用者要走 `完整 SA 流程` 或 `輕量開發文件`。選輕量則停下來導引使用者改調用 `codebase-dev-doc-builder`，本 skill 不再往下產文件；選完整則續行。
3. 若為修訂現有 SA 文件，先讀取現有的 `.md` 檔案。
4. 判斷目標是既有專案或全新專案。
5. **既有專案**：讀取 `references/codebase-discovery.md`，透過 GitNexus 查找與 PRD 相關的現有程式碼、組件、流程、角色與規則，並在查找的同時推進文件設計。**全新專案**：直接進入文件設計。
6. 區分四種來源：PRD 確認的需求、程式碼確認的現況、合理推論、以及本次 SA 為收斂所做的架構判斷；尚未確認者標記為待釐清。
7. 過程中一旦出現「核心行為規範」所列的任何疑問，立即停下來向使用者提問，不要帶著假設繼續。
8. 完成初步分析後，依「模組／功能分解」用選項反問使用者，決定輸出為單一 SA 或拆分為母文件＋多模組 SA。
9. 讀取 `references/sa-document-template.md`，依定案結構整理：單一時產出一份 SA；拆分時產出母文件（拆分總覽）＋各模組 SA，保留來源標記。
10. 把所有曾被跳過、暫緩、或尚未達成共識的問題彙整到「待釐清事項」章節（拆分時，整體層級問題放母文件，模組層級問題放各模組 SA）。
11. 將完成的文件寫入 `.md` 檔案（拆分時一併寫出母文件與各模組 SA）。
12. 回報輸出路徑（拆分時列出母文件與各模組檔案）、文件內容的簡短摘要，以及目前「待釐清事項」狀態。

## Codebase Discovery（既有專案）

除非使用者明確表示為全新專案或無需查找，否則對既有專案一律透過 GitNexus 查找相關程式碼，遵循 `references/codebase-discovery.md`。

- 以程式碼發現確認現況的組件、職責、流程、角色與業務規則，作為 `Code-confirmed` 依據。
- 程式碼發現以**架構影響**為框架：目前由哪些組件承載、可複用的能力、現有邊界與限制、與 PRD 的落差。
- 不臆造需求、權限、流程或規則；所有陳述須有 PRD、討論或程式碼依據。
- 不將 SA 文件變成實作清單；逐方法、逐檔案的細節不放入 SA。

## Source Markers

在分析過程與最終文件中一致使用以下來源標記，確保每個陳述可追溯。原有標記完整保留，並新增 `Design-decision` 以標示本次為收斂架構所做的判斷：

| 標記 | 意義 |
| --- | --- |
| PRD-confirmed | 來自 PRD（文件、網址、Issue、需求討論或使用者補充）。 |
| Code-confirmed | 來自既有專案程式碼（多經 GitNexus 查找確認）。 |
| Inferred | 依 PRD 與程式碼合理推導，尚未明確確認。 |
| Design-decision | 本次 SA 為收斂架構主動做出的判斷，可被後續審查推翻。 |
| Unknown | 尚未確認、需釐清的問題。 |

結構化內容優先使用名為 `來源` 的表格欄位，而非在每個句子前重複標記；敘事段落僅在有助於理解時，才以短前綴（如 `PRD-confirmed：`）標示。若使用者明確要求移除來源標記，才省略。

## Clarification & Open Questions

遵循 `references/clarification-and-open-questions.md`。

- 高影響力或邊界不清的疑問：依「核心行為規範」立即停下來提問，不得擅自假設。
- 經提問後被暫緩、跳過、或尚未達成共識的問題：彙整至「待釐清事項」章節。
- SA 文件中必須保留「待釐清事項」章節。修訂時，把已回答的問題移至對應的需求、規則、流程或架構章節，再從「待釐清事項」移除該列；若無未解問題，保留章節並填入一列 `目前無待釐清事項`。

## Document Goal

產出一份架構層級的 SA 文件，適合需求審查、利害關係人討論、範圍確認與迭代釐清，並足以讓後續 AI 僅依此文件獨立完成 SD。

文件依據：
- 提供的 PRD 內容與相關討論
- 既有專案程式碼的實際現況（經 GitNexus 查找）
- 明確的限制、假設、依賴與未解問題

## Content Requirements

遵循 `references/sa-document-template.md`。最終 SA 文件預設結構：

1. `文件資訊`
2. `來源標記說明`
3. `一、功能概述`
4. `二、角色與權限`
5. `三、範圍`
6. `四、需求與業務規則`（FR 功能需求 + BR 業務規則）
7. `五、流程`
8. `六、架構概觀`
9. `七、資料與整合概念`
10. `八、待釐清事項`

章節並非僵硬規定；可依 PRD 性質與專案情境，以對 SA 的理解主動調整最合適的呈現方式，但須維持架構層級、來源可追溯，且保留「待釐清事項」。SA 維持架構層級，不含驗收條件；驗收條件由下游 `codebase-sa-sd-finalizer` 在 `-sd.md` 補上。

**拆分時的結構**：母文件採「拆分總覽 SA」結構（見 `references/sa-document-template.md`），承載整體功能概述、整體範圍、**模組／功能分解**（各模組範圍、彼此依賴與先後順序、對應檔名）、跨模組業務規則與共用概念、整體層級待釐清事項，以及**跨模組／端對端驗收條件**（沒有任何單一模組能獨佔的）。各模組 SA 採上述標準 SA 結構，聚焦該模組，並在 `文件資訊` 指回母文件。母文件不往下游 finalize，故跨模組／端對端驗收條件只能放母文件這裡；模組層級的驗收條件不放母文件，也不放模組 SA，而是由下游 SD 補上。

## Output Quality Bar

文件必須符合：
- 可作為獨立的 `.md` 檔案閱讀，繁體中文撰寫。
- 嚴格維持架構層級，不含應屬 SD 的設計細節，且不出現任何「由 SD 設計／供 SD 參考」式的轉嫁語句。
- 以來源標記區分確認事實、推論、架構判斷與未知項目；結構化內容優先使用 `來源` 表格欄位。
- 在有助於提升可讀性的情境下，使用 Markdown 表格整理資訊，例如範圍、角色、欄位、業務規則、架構組件與待釐清事項等。流程以 Mermaid flowchart 呈現主線、異常路徑以條列補充；架構以 Mermaid graph 呈現組件關係與互動。
- 功能包含多個頁面、流程或能力時，以 `FR-01`、`FR-02` 等結構化功能需求群組呈現；業務規則以 BR 編號並說明違反時的處理。
- 架構概觀清楚交代受影響的組件與職責、彼此互動、邊界與整合點，使下游 SD 能據以展開設計。
- 必定保留「待釐清事項」；無未解問題時才填入 `目前無待釐清事項`。
- 拆分時，母文件須保留「跨模組／端對端驗收條件」，並**精簡收斂**：僅列核心、明確可判斷、沒有任何單一模組能獨佔的項目，以「夠用即止」為原則。

## Execution Constraint

寫入 Markdown 檔案後：
- 停止。
- 僅回報檔案路徑、文件內容的簡短摘要，以及目前「待釐清事項」狀態。

若使用者之後要求修訂文件：
- 先讀取現有的 `.md` 檔案與使用者新回饋。
- 僅針對修訂涉及的範圍重新透過 GitNexus 檢視程式碼。
- 除非使用者要求新檔案，否則就地更新現有文件。
- 保留先前已確認的內容，除非新回饋或程式碼檢查有所矛盾。
