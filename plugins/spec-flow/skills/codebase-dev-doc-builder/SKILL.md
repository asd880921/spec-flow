# codebase-dev-doc-builder

針對**規模小、邊界明確**的需求，產出**一份**輕量的「開發文件」`<feature>-dev.md`。這是完整 `codebase-sa-builder` + `codebase-sa-sd-finalizer` 雙階段流程之外的**第三條輕量路徑**：當需求小到不需要架構層級的對齊、但仍需要一份可追溯、可驗收的書面依據時使用。

本 skill 一次產出單一文件，**不**做模組拆分反問、**不**做「先複製原始文件再收斂」的雙文件儀式、**不**強制 Mermaid 架構圖。產出的 `-dev.md` 可直接交給 `dev-planner` 規劃 commit，最後一章的驗收條件供 `dev-acceptance-verifier` 逐條驗收。

## 適用邊界（與完整 SA 流程的分流）

本 skill 適用於**三個判準皆否**的需求：

1. 是否會新增或調整**角色與權限**？
2. 是否會跨越目前**單一模組的職責邊界**（牽動其他模組）？
3. 是否需要新建或調整**既有架構假設**？

- **三者皆否** → 適用本 skill（小功能增修、欄位調整、單一規則修改、侷限單一模組的行為調整等）。
- **符合任一項** → 不適用；停下來導引使用者改用 `codebase-sa-builder` 走完整 SA 流程（不自動調用）。

若被調用後才發現需求其實觸及上述任一面向（規模被低估），**立即停下來**告知使用者，建議改走完整 SA 流程，不要勉強用輕量文件硬撐架構層級的決策。

## 核心行為規範（最高優先，調用時必須遵守）

**遇到模糊需求必須暫停並反問，不得擅自假設或繼續設計。**

- 需求不明確、語意有多種解讀、或與現有程式碼矛盾時，立即停下來提問。
- 判斷需求其實超出「適用邊界」時，停下來導引回 `codebase-sa-builder`。
- 經提問後被暫緩、跳過或尚未達成共識的問題，必須完整彙整至「待釐清事項」章節，不得遺漏。

## Input

請提供下列其中一項作為需求來源：
- 直接在 prompt 中提供的小需求內容。
- 需求文件、規格或其節錄。
- Issue 內容（可直接貼上，或提供 `project_id` 與 `issue_number` 以透過 `gitlab-issue-fetch` 取得）。
- 待修訂的現有 `-dev.md` 開發文件。

選填：是否為既有專案 / 全新專案、輸出檔名、輸出目錄、語言覆蓋。

未提供輸出路徑時：
- 寫入目前工作目錄。
- 有 issue 編號時使用 `issue-<issue_number>-dev.md`；否則使用 `<feature>-dev.md`，無法判斷 feature 時使用 `dev.md`。

預設輸出語言為繁體中文；釐清問題預設以繁體中文詢問。

## Workflow

1. 取得並通讀需求。內容不完整且提供了 `project_id` 與 `issue_number` 時，先呼叫 `gitlab-issue-fetch`。
2. 若為修訂現有 `-dev.md`，先讀取現有檔案。
3. 對照「適用邊界」三判準確認規模；若超出邊界，停下來導引回 `codebase-sa-builder`。
4. 既有專案：依下方「Codebase Discovery（簡化版）」透過程式碼知識圖譜查找可沿用的既有實作與合理的落點。全新專案：直接設計。
5. 過程中一旦出現「核心行為規範」所列疑問，立即停下來提問。
6. 讀取 `references/dev-doc-template.md`，依其結構整理出一份精簡的 `-dev.md`，保留來源標記與「待釐清事項」，並在最後一章列出驗收條件。
7. 把所有曾被跳過、暫緩或尚未達成共識的問題彙整到「待釐清事項」章節。
8. 將文件寫入 `-dev.md`。
9. 回報輸出路徑、文件內容的簡短摘要，以及目前「待釐清事項」狀態。

## Codebase Discovery（既有專案，簡化版）

既有專案優先透過程式碼知識圖譜查找，但**只查與本次小改動直接相關的點**，不展開完整架構影響分析：

- 既有是否已有可沿用的實作、元件或模式，避免重造。
- 本次調整最合理的落點（既有哪個檔案 / 模組 / 層級承載相關職責）。
- 既有命名、資料結構、驗證或錯誤處理慣例，使後續 commit 與現況一致。

查得的現況以 `Code-confirmed` 標記。程式碼知識圖譜查找不可用或查無結果時，退用一般搜尋與檔案閱讀，仍只擷取與本次改動直接相關的事實。

## Source Markers

在文件中一致使用以下來源標記，確保可追溯：

| 標記 | 意義 |
| --- | --- |
| PRD-confirmed | 來自需求（文件、Issue、討論或使用者補充）。 |
| Code-confirmed | 來自既有專案程式碼（多經程式碼查找確認）。 |
| Inferred | 依需求與程式碼合理推導，尚未明確確認。 |
| Design-decision | 本次為收斂方向主動做出的判斷，可被審查推翻。 |
| Unknown | 尚未確認、需釐清的問題。 |

結構化內容優先使用名為 `來源` 的表格欄位。若使用者明確要求移除來源標記，才省略。

## Content Requirements

遵循 `references/dev-doc-template.md`。`-dev.md` 預設結構：

1. `文件資訊`
2. `來源標記說明`
3. `一、需求概述`
4. `二、範圍`
5. `三、需求與規則`（簡要 FR / BR）
6. `四、受影響範圍（現況與調整）`
7. `五、待釐清事項`
8. `六、驗收條件`（最後一章）

章節並非僵硬規定，可依需求性質精簡或合併，但須保留「待釐清事項」與最後一章「驗收條件」，且維持來源可追溯。**保持輕量**：小需求的章節以一兩列表格或短段落帶過即可，不為填滿模板而膨脹。

## Output Quality Bar

文件必須符合：
- 可作為獨立的 `.md` 檔案閱讀，繁體中文撰寫，篇幅與需求規模相稱（小需求就應該短）。
- 不做架構層級的展開（角色權限體系、跨模組互動、架構圖）；若發現確實需要，代表規模被低估，應導引回 `codebase-sa-builder`。
- 以來源標記區分確認事實、推論、判斷與未知；結構化內容優先使用 `來源` 表格欄位。
- 必定保留「待釐清事項」；無未解問題時填入 `目前無待釐清事項`。
- **最後一章為驗收條件**，列核心、明確可判斷的項目，精簡收斂、夠用即止；此章供下游 `dev-acceptance-verifier` 逐條驗收。

## Execution Constraint

寫入 `-dev.md` 後：
- 停止。
- 僅回報檔案路徑、文件內容的簡短摘要，以及目前「待釐清事項」狀態。

若使用者之後要求修訂：
- 先讀取現有的 `-dev.md` 與新回饋。
- 僅針對修訂涉及的範圍重新查找程式碼。
- 除非使用者要求新檔案，否則就地更新現有文件。

## 下一步引導（純提示，不主動調用）

完成後主動告知使用者後續選項，但**不自行調用**下一個 skill：
- **順流下一步**：將 `-dev.md` 交給 `dev-planner` 規劃 atomic commits（`dev-planner` 接受 `-dev.md` 作為任務來源），再由 `dev-builder` 逐 commit 執行，最後以 `dev-acceptance-verifier` 驗收 `-dev.md` 最後一章的驗收條件。
- **回頭修正 / 升級**：若開發中發現需求其實涉及架構、角色權限或跨模組，停下來導引使用者改走 `codebase-sa-builder` 完整 SA 流程。
- **交付清洗（非順流步驟，由使用者自行決定）**：若想產出移除內部註記、可對外交付的乾淨版，可自行調用 `codebase-design-doc-cleaner` 清洗當下文件。它是隨時可用的工具、可在文件多次修訂的過程中重複使用，非固定流程步驟，本 skill 不主動引導使用者進入。

完整流程見 `README.md`。
