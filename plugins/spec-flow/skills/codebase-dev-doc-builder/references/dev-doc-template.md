# 開發文件 Template

起草或修訂輕量開發文件 `-dev.md` 時，請使用本模板。本文件服務「規模小、邊界明確」的需求，同時供開發對照與驗收使用，因此精簡好讀、且足以讓 `dev-planner` 接手規劃 commit、`dev-acceptance-verifier` 逐條驗收。

## Required Style

- 預設以繁體中文撰寫。
- **保持輕量**：篇幅與需求規模相稱。小需求的章節以一兩列表格或短段落帶過即可，不為填滿模板而膨脹。
- **不**做架構層級的展開：角色權限體系、跨模組互動、架構圖等。若發現確實需要，代表規模被低估，應停下來導引回 `codebase-sa-builder`。
- 保留來源標記，讓讀者理解每個陳述的依據。一致使用：`PRD-confirmed`、`Code-confirmed`、`Inferred`、`Design-decision`、`Unknown`。
- 結構化內容優先使用含 `來源` 欄位的 Markdown 表格，避免在有表格來源欄時仍逐句重複標記。
- `來源標記說明` 緊接 `文件資訊` 之後。
- 除標點符號外，所有特殊符號（`/`、`+`、`-`、`...` 等）一律使用半形。
- 必須保留 `待釐清事項`；若無未解問題，保留章節並填入一列 `目前無待釐清事項`。
- 最後一章必為 `驗收條件`。

## Markdown Structure

```text
# <功能名稱> 開發文件

## 文件資訊

| 項目 | 內容 |
| --- | --- |
| 功能名稱 | <功能名稱> |
| 需求來源 | <文件 / 網址 / Issue（project_id, issue_number）等> |
| 專案類型 | <既有專案 / 全新專案> |
| 文件狀態 | <草稿 / 審查中 / 定稿> |

## 來源標記說明

| 標記 | 意義 |
| --- | --- |
| PRD-confirmed | 來自需求（文件、Issue、討論或使用者補充）。 |
| Code-confirmed | 來自既有專案程式碼（多經 GitNexus 查找確認）。 |
| Inferred | 依需求與程式碼合理推導，尚未明確確認。 |
| Design-decision | 本次為收斂方向主動做出的判斷，可被審查推翻。 |
| Unknown | 尚未確認、需釐清的問題。 |

## 一、需求概述

<用 1-2 段說明這次要做什麼、為什麼、預期結果。需求單純時一段即可。>

| 來源 | 目標 | 說明 |
| --- | --- | --- |
| PRD-confirmed | <目標> | <說明> |

## 二、範圍

| 來源 | 類型 | 項目 | 說明 |
| --- | --- | --- | --- |
| PRD-confirmed | 包含 | <項目> | <說明> |
| PRD-confirmed | 不包含 | <項目> | <說明> |

## 三、需求與規則

> 功能需求（FR）描述要做什麼；業務規則（BR）描述約束與違反時的處理。需求單純時可只列其一，或合併為數列。

| 來源 | 類型 | 項目 | 規格 / 規則 | 違反時的處理 |
| --- | --- | --- | --- | --- |
| PRD-confirmed | FR | <功能項目> | <規格> | 無 |
| PRD-confirmed | BR | <規則描述> | <條件> | <錯誤回應 / 阻擋 / 警告等> |

## 四、受影響範圍（現況與調整）

> 簡述本次會動到哪些既有組件 / 檔案 / 層級，以及預計怎麼調整。既有專案以 GitNexus 簡化查找的現況為依據；不畫架構圖、不展開跨模組互動。

| 來源 | 組件 / 區塊 | 現況 | 本次調整 |
| --- | --- | --- | --- |
| Code-confirmed | <現有組件或檔案> | <目前行為 / 可沿用的能力> | <預計新增或修改的內容> |

## 五、待釐清事項

| 來源 | 問題 | 影響範圍 | 建議處理方式 |
| --- | --- | --- | --- |
| Unknown | <待釐清事項；若沒有，填寫「目前無待釐清事項」> | <影響範圍；若沒有，填寫「無」> | <建議由誰回答或後續如何處理；若沒有，填寫「無」> |

## 六、驗收條件

整份文件的驗收條件，必為最後一章。列核心、明確可判斷的項目，精簡收斂、夠用即止，不膨脹成測試案例清單。供下游 `dev-acceptance-verifier` 逐條驗收。

| 編號 | 驗收條件 |
| --- | --- |
| AC-01 | <核心、可明確判斷的驗收條件> |
```

## Section Guidance

- `文件資訊`：metadata，保持簡潔。
- `來源標記說明`：除非使用者明確要求移除，否則始終置於文件前段。
- `需求概述`：說明要做什麼、為什麼、預期結果；需求單純時一段帶過。
- `範圍`：明確區分包含與不包含；小需求常常一兩列就夠。
- `需求與規則`：FR 與 BR 合併在同一表，以 `類型` 欄位區分；需求單純時可只列必要列。
- `受影響範圍（現況與調整）`：本文件供 `dev-planner` 接手的關鍵章節。簡述受影響的既有組件、現況與預計調整，使 commit 規劃有對照依據；不畫架構圖、不做跨模組互動分析（需要時代表規模被低估，應回 `codebase-sa-builder`）。
- `待釐清事項`：必須保留。修訂時把已回答列移至對應章節並更新來源標記；無剩餘時保留一列 `目前無待釐清事項`。
- `驗收條件`：必為最後一章。精簡收斂，僅列核心且可明確判斷的項目，預設不加 `來源` 欄（來源可從前述章節追溯）。
