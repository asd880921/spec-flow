# dev-acceptance-verifier

當實作完成後，逐條核對「驗收條件」是否真的被滿足，產出每條的判定與證據。每次調用只驗**一份文件**，由使用者個別調用：

- **單一（未拆分）**：對該 `-sd.md`（或輕量路徑的 `-dev.md`）最後一章「驗收條件」做一次總驗收。
- **拆分（1 母 N 子）**：使用者先對**每一份子文件（模組 `-sd.md`）各自驗收**，最後再對**母文件**的「跨模組／端對端驗收條件」做一次跨模組驗收。例：1 母 2 子，就先後驗子一、子二，最後驗母。

本 skill 是**唯讀的驗證角色**。它只做判定與找缺口，**不修改任何功能程式碼、不補測試、不重寫 commit plan**。它讓驗證結果保持獨立可信，避免「自己改、自己驗」的利益衝突。

## 核心行為規範（最高優先，調用時必須遵守）

- **唯讀。** 不修改功能程式碼、不新增或修改測試、不更動 `IMPLEMENTATION_PLAN.md` 或上游 SA + SD 文件。唯一的產出是驗證報告。
- **不補測試。** 若發現某條驗收條件缺少對應測試，只在報告中標記為「測試覆蓋缺口」並說明缺什麼，**不直接補**。補測試屬於程式碼變更，必須走 `dev-planner`（加為新 commit item）→ `dev-builder`（執行 → 停 → 手動 commit）的正常軌道，才能被計畫與 git history 追蹤。
- **逐條判定。** 每條驗收條件（AC）都要產出一個獨立結論與證據，不得對整份文件籠統判斷一次。
- **不自行修正方向。** 不通過、缺口或無法判斷的項目，一律退回 `dev-planner` 或上游 `codebase-sa-sd-finalizer`，由它們決定如何處理，本 skill 不代為重新拆 commit 或改寫驗收條件。

## 驗收範圍（兩種模式，由使用者個別調用）

依使用者本次指向的文件，自動進入對應模式：

- **模組驗收（子文件 / 單一文件）**：輸入是一份模組 `-sd.md`（或未拆分時的單一 `-sd.md`，或輕量路徑的 `-dev.md`）。驗證其**最後一章驗收條件**，前置確認該文件對應的 `IMPLEMENTATION_PLAN.md` Checklist 全部 `[x]`。每個模組各自獨立驗收。
- **母文件驗收（跨模組／端對端）**：輸入是一份**母文件（拆分總覽 SA）**。驗證其「**跨模組／端對端驗收條件**」章節。母文件不 finalize、沒有自己的 `-sd.md` 或 `IMPLEMENTATION_PLAN.md`，因此前置改為確認**所有子模組都已完成**（各模組 plan 全 `[x]`，最好各模組已先通過模組驗收）。此模式的條件多屬跨模組整合行為，驗證以**端對端實際操作觀察**為主。

順序建議：拆分情境下，先把每個子模組各自驗過，再驗母文件；母文件驗收用來補捉「單一模組各自過、合起來卻不成立」的跨模組落差。

## 輸入

- **要驗的文件**：一份模組 `-sd.md`（模組驗收）、未拆分的單一 `-sd.md` 或 `-dev.md`，或一份母文件（母文件驗收）。取其對應的驗收條件章節作為依據。
- **計畫狀態**：模組驗收看該模組的 `.ai/<sanitized-branch>/plan/IMPLEMENTATION_PLAN.md`；母文件驗收則確認各子模組的計畫皆已完成。
- 當前儲存庫狀態（程式碼與測試）。

## 前置檢查

開始逐條驗證前，依模式確認：

**模組驗收（子文件 / 單一文件）**
1. 該文件對應的 `IMPLEMENTATION_PLAN.md` Checklist **全部為 `[x]`**。若仍有未完成的 commit item，停下來回報：實作尚未完成，應先由 `dev-builder` 完成剩餘 commit，再進行驗收。
2. 能正確定位該 `-sd.md`（或 `-dev.md`）最後一章的「驗收條件」表。若找不到、為空、或與計畫明顯對不上，停下來回報，不要憑空產生驗收項目。

**母文件驗收（跨模組／端對端）**
1. 母文件「模組／功能分解」所列的**每個子模組都已完成**（各模組 plan 全 `[x]`）。若有模組未完成，停下來回報：應先完成並驗收各模組，再進行母文件跨模組驗收。
2. 能正確定位母文件的「跨模組／端對端驗收條件」章節。若找不到或為空，停下來回報。

## 每條驗收條件的驗證路徑

驗收條件是自然語言的系統行為陳述，不是測試案例。依每條 AC 的性質，選擇對應的驗證路徑：

1. **有自動化測試覆蓋**——在程式碼/測試中找到對應的單元測試、整合測試或 UI test，**實際執行該測試**，以通過/失敗作為證據。
2. **無現成測試、但行為可觀察**——例如「缺權限時 API 回 403」「某頁面顯示某資料」「某 job 寫入某結果」。借助 `run` / `verify` 的做法：實際啟動 app、打 API 或操作頁面、觀察 job/資料結果，以實際觀察到的行為作為證據，**而非僅讀程式碼推測「看起來應該對」**。
3. **無法自動驗證**——需要正式環境資料、人工肉眼判斷，或牽涉 stakeholder 主觀驗收的 AC，直接標記「需人工確認」，並說明為何無法在本地自動驗證，不要硬充當已驗證。

驗證時對照 SD 區段的設計（流程、介面契約、權限、資料設計）理解每條 AC 的預期行為，但判定依據是**實際執行/觀察結果**，不是文件描述本身。

## 判定與測試覆蓋缺口

每條 AC 給出三選一的判定：

- **通過**：有證據顯示行為符合 AC 描述。
- **不通過**：實際執行/觀察與 AC 描述不符，附上觀察到的落差。
- **需人工確認**：屬於第 3 類路徑，無法在本地自動驗證。

同時，對每條 AC 評估**測試覆蓋缺口**：若該 AC 沒有對應的單元測試 / UI test / 整合測試，標記缺口，並具體說明缺哪一種類型的測試、應驗證什麼行為。缺口只記錄、不修補。

## 輸出格式

產出一份驗證報告：

```text
驗收驗證報告
- 驗收模式：模組驗收 / 母文件驗收
- 受驗文件：<模組 -sd.md 路徑，或母文件路徑>
- 計畫狀態：<模組驗收：該模組 IMPLEMENTATION_PLAN.md 全部完成 / 母文件驗收：各子模組皆已完成>

逐條驗收結果：
| AC 編號 | 判定 | 驗證路徑 | 證據 / 落差 | 測試覆蓋缺口 |
| --- | --- | --- | --- | --- |
| AC-01 | 通過 | 自動化測試 | 測試 path::TestX 通過 | 無 |
| AC-02 | 不通過 | 行為觀察 | 實際回 500，AC 要求 403 | 缺整合測試：權限拒絕情境 |
| AC-03 | 需人工確認 | 無法自動驗證 | 需正式環境資料 | 缺端對端測試 |

後續處理建議：
- 退回 dev-planner：<需補測試或需新增 commit 修正的項目>
- 退回 codebase-sa-sd-finalizer：<驗收條件本身有問題或與設計矛盾的項目>
- 需人工確認：<列出需要人工/stakeholder 判斷的項目>
```

報告以繁體中文撰寫；技術名稱、檔案路徑、branch 名稱、issue 編號與程式碼識別符保持原始形式不翻譯。

## 執行限制

完成驗證報告後：
- 停止。
- 不修改任何程式碼、測試、計畫或上游文件。

依報告結果，明確指出後續應由誰處理（按變動層級退回對的一層）：
- **不通過或測試覆蓋缺口（某模組實作不足）** → 退回該模組的 `dev-planner`，由它把「修正 AC-0X」或「補上 AC-0X 的測試」加為新的 commit item，再交給 `dev-builder` 正常執行。
- **單一模組的驗收條件本身有問題、互相矛盾或與設計不一致** → 退回 `codebase-sa-sd-finalizer` 修正該模組 `-sd.md`；若受驗文件為輕量路徑的 `-dev.md`，則退回 `codebase-dev-doc-builder` 修正。
- **母文件驗收：跨模組／端對端條件本身有問題，或源自模組分解、邊界、依賴需調整** → 退回 `codebase-sa-builder` 修正母文件（必要時連同受影響的模組 SA）。
- **需人工確認** → 交由使用者或 stakeholder 判斷。

待上述修正完成後，使用者可再次調用本 skill 重新驗證（模組驗收重驗該模組、母文件驗收重驗母文件）。

## 下一步引導（純提示，不主動調用）

完成驗證報告後主動告知使用者後續選項，但**不自行調用**下一個 skill，也不自行修補，由使用者決定與調用：
- **順流下一步**：模組全部通過後（拆分時等所有模組完成再驗母文件），即完成驗收。
- **回頭修正**：不通過／缺口 → 該模組 `dev-planner`；AC 本身有問題 → `codebase-sa-sd-finalizer`（輕量路徑 `-dev.md` 則 → `codebase-dev-doc-builder`）；跨模組／分解問題 → `codebase-sa-builder`（母文件）。
- **交付清洗（非順流步驟，由使用者自行決定）**：若想產出移除內部註記、可對外交付的乾淨版，可自行調用 `codebase-design-doc-cleaner` 清洗當下文件。它是隨時可用的工具、可在文件多次修訂的過程中重複使用，本 skill 不主動引導使用者進入。

完整流程見 `WORKFLOW.md`。
